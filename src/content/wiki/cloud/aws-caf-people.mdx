## 정의

**People Perspective** 는 [[aws-caf|AWS CAF]] 6 관점 중 **비즈니스 와 기술 사이의 다리** 역할을 하는 관점입니다. **문화, 조직 구조, 리더십, 인력 (workforce)** 을 다루며, "변화 (change) 가 일상 (business-as-normal) 이 되는" 조직 문화로의 진화를 촉진합니다.

**주요 stakeholder**: CIO, COO, CTO, cloud director, cross-functional/enterprise-wide leaders.

## 왜 People 관점이 중요한가

기술적으로 성숙한 조직도 **사람 관점 미숙** 이면 실패:

- 스킬 갭 -> 프로덕션 문제
- 저항 문화 -> 도입 지연
- 사일로 조직 -> 협업 저해
- 리더십 부재 -> 방향성 상실

**"기술은 쉽고, 사람이 어렵다"** 는 격언이 여기 해당.

## 주요 Capabilities

### 1. Culture Evolution (문화 진화)

**실험, 학습, 실패 허용** 문화로 진화. 위계적 승인 -> 자율적 결정.

**측정**:
- 실험 횟수/분기
- Blameless postmortem 실행
- 팀 자율성 지수

### 2. Transformational Leadership (변혁적 리더십)

경영진의 클라우드 전환 명확한 커밋. 리더 자체가 클라우드/디지털에 대한 이해 필수.

**활동**:
- 리더 클라우드 교육 (AWS Cloud Practitioner 등)
- 정기 town hall 에서 비전 반복
- 성공 스토리 공유

### 3. Cloud Fluency (클라우드 유창성)

**모든 직원이 클라우드 기초 이해**. IT 팀만이 아니라 재무/HR/영업도.

- Cloud 101 교육
- 용어집 (glossary)
- 사내 뉴스레터

### 4. Workforce Transformation (인력 전환)

**스킬 갭 매핑 + 재교육 계획**.

- 현재 스킬 인벤토리
- Target 스킬 매핑
- Learning path 별 교육
- AWS Certifications (Cloud Practitioner, Solutions Architect Associate, ...)
- 외부 채용 vs 내부 재교육

### 5. Organizational Design (조직 설계)

**클라우드 지향 팀 구조**. Product team, DevOps team, Cloud Center of Excellence (CCoE).

**전통**:
```
Dev team -----> QA team -----> Ops team -----> Prod
```

**클라우드 지향**:
```
Product team (Dev + QA + Ops)
   ↓
CCoE (표준, 도구, 지원)
```

### 6. Organizational Alignment (조직 정렬)

여러 팀이 같은 목표 / 인센티브. 사일로 감소.

### 7. Team Enablement (팀 인에이블먼트)

**셀프서비스 도구, 문서, 지원**. 팀이 자율적으로 클라우드 활용.

- Cloud portal (IaC 템플릿, 표준)
- 사내 Slack channel (`#cloud-help`)
- Office hours

### 8. Continuous Learning (지속 학습)

기술은 계속 변함. 학습이 상시 프로세스.

- Learning budget 개인/팀별
- Conference/summit 참가
- Book club, tech talk
- Certification renewal

## Cloud Center of Excellence (CCoE)

People perspective 의 핵심 조직 단위. **클라우드 관련 전문성 + 표준 + 지원** 을 담당.

### 책임

- Landing Zone / 계정 관리
- 표준 (naming, tagging, security)
- Reusable IaC 모듈
- 교육 프로그램
- 팀 지원 (컨설팅, 페어링)
- FinOps 리더십

### 조직 배치

**Federated CCoE**: 중앙 팀 + 각 사업부 embed. 규모 있는 조직.

**Central CCoE**: 하나의 중앙 팀. 시작 단계.

**Hub-and-spoke**: 중앙 CCoE + 각 팀의 champion. 균형.

## AWS Certifications Path

**Foundational**:
- AWS Certified Cloud Practitioner (모든 직원 권장)

**Associate** (기술직):
- Solutions Architect Associate
- Developer Associate
- SysOps Administrator Associate

**Professional** (시니어):
- Solutions Architect Professional
- DevOps Engineer Professional

**Specialty** (전문):
- Security, Advanced Networking, Machine Learning, Data Analytics, Database

**AI 시리즈** (2024+):
- AWS Certified AI Practitioner
- AWS Certified Machine Learning Engineer Associate
- AWS Certified Machine Learning Specialty

## 실전 활동

### Envision

- 조직 문화 진단 (survey)
- 현재 스킬 인벤토리
- 리더십 커밋 확보

### Align

- CCoE 구조 설계
- 팀별 target skill 매핑
- Learning path 정의

### Launch

- 첫 팀 재교육 (pilot)
- CCoE 시범 운영
- 문화 이니셔티브 (postmortem, tech talk)

### Scale

- 전사 확장
- 지속 인에이블먼트 (매년 certification renewal)
- 채용 브랜딩 (클라우드 인재)

## 문화 변화 사례

### DevOps 문화

- Dev + Ops 팀 통합
- "You build it, you run it"
- Blameless postmortem

### 실험 문화

- Two-pizza team
- Feature flag 로 A/B
- 실패는 학습 기회

### Data-driven 문화

- 결정을 데이터로
- Dashboard 상시 참조
- 가설 -> 실험 -> 검증

## 흔한 실패 패턴

> [!WARNING]
> **교육 없이 도입**. IT 팀이 EC2, S3 밖에 몰라 클라우드 네이티브 서비스 (Lambda, EKS, Bedrock) 활용 못 함.

> [!CAUTION]
> **CCoE 를 통제 조직으로**. Enabler 여야 하는데 "허가 게이트" 로 전락하면 팀 자율성 저해.

> [!WARNING]
> **문화 변화는 몇 년 걸림**. 기술 도입 (몇 개월) 대비 훨씬 오래. 인내와 리더십 지속.

> [!IMPORTANT]
> **리더가 first mover**. 임원 교육을 먼저 (Cloud Practitioner 등). 모범 없이 아래 요구 안 됨.

> [!CAUTION]
> **인재 유출 리스크**. 재교육받은 인재를 경쟁사가 스카웃. 커리어 트랙 + 대우 정비.
