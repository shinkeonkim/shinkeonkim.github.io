## 정의

**Flask** 는 Armin Ronacher 가 2010년 발표한 **WSGI 기반 Python 마이크로프레임워크** 입니다. Pallets Projects 가 유지관리하고, 2026년 현재 **Flask 3.x + Werkzeug 3.x + Jinja2 3.x** 조합이 표준입니다.

핵심 철학: "**작은 코어 + 유연한 확장**". 라우팅, 요청/응답, 템플릿 렌더링만 코어에 두고 DB, 인증, 폼, 마이그레이션 등은 extension 으로 조합.

## 왜 Flask 인가

- **최소 진입**: 5줄로 서버 실행
- **명시적**: 마술 (auto-discovery, convention over configuration) 이 적어 디버깅 쉬움
- **거대한 생태계**: 20년 축적된 extension, 스택오버플로 자료
- **테스트 용이**: `test_client()` 로 앱 인스턴스 직접 테스트
- **Jinja2**: SSR 강력, HTMX/Alpine 조합에 잘 맞음

## 핵심 개념

### WSGI

**Web Server Gateway Interface** (PEP 3333). Sync 인터페이스. `def app(environ, start_response) -> Iterable[bytes]`. Flask 인스턴스가 WSGI 앱.

FastAPI (ASGI) 와 근본적으로 다름: WSGI 는 async 지원이 제한적이고 한 프로세스가 요청 하나만 처리.

### App Context / Request Context

Flask 는 thread-local (실제로는 `LocalStack`) 을 사용해 **request context** 와 **application context** 를 관리. 전역처럼 보이는 `request`, `session`, `current_app`, `g` 는 사실 스택 위 객체 프록시.

```python
from flask import request, session, current_app, g

@app.get("/me")
def me():
    print(request.args)          # query params
    print(session.get("user"))   # session
    print(current_app.config)    # 앱 config
    g.request_id = "..."         # request-scoped 저장소
    return "..."
```

- **`request`**: 현재 요청 (path, method, args, form, files, headers, cookies)
- **`session`**: 클라이언트 sessions (기본은 서명 쿠키)
- **`current_app`**: 현재 앱 인스턴스 (blueprint 안에서 유용)
- **`g`**: 요청 스코프 임시 저장소 (DB session 등)

### Extension

Flask 는 코어에 없는 기능을 extension 으로 제공. `Flask-SQLAlchemy`, `Flask-Login`, `Flask-Migrate`, `Flask-WTF`, `Flask-CORS`, `Flask-Caching`, ...

## 프로젝트 구조 (실전)

```
myapp/
├── src/
│   └── myapp/
│       ├── __init__.py       # create_app()
│       ├── config.py         # Config class
│       ├── extensions.py     # db = SQLAlchemy(), login = LoginManager(), ...
│       ├── models.py         # SQLAlchemy models
│       ├── auth/             # Blueprint
│       │   ├── __init__.py
│       │   ├── routes.py
│       │   └── forms.py
│       ├── main/
│       │   └── routes.py
│       └── templates/
│           └── base.html
├── migrations/               # Flask-Migrate (Alembic)
├── tests/
├── pyproject.toml
└── wsgi.py                   # 진입점
```

### create_app (Application Factory)

```python
# myapp/__init__.py
from flask import Flask
from myapp.extensions import db, migrate, login

def create_app(config_name: str = "production") -> Flask:
    app = Flask(__name__)
    app.config.from_object(f"myapp.config.{config_name.capitalize()}Config")

    db.init_app(app)
    migrate.init_app(app, db)
    login.init_app(app)

    from myapp.auth.routes import bp as auth_bp
    from myapp.main.routes import bp as main_bp
    app.register_blueprint(auth_bp, url_prefix="/auth")
    app.register_blueprint(main_bp)

    return app
```

자세한 것은 [[flask-app-factory|Flask App Factory]] 참조.

## Request 처리

```python
from flask import request

@app.post("/echo")
def echo():
    # JSON body
    data = request.get_json()

    # Form data
    name = request.form.get("name")

    # Query
    q = request.args.get("q", default="")

    # 파일
    f = request.files.get("upload")
    if f:
        f.save(f"/tmp/{f.filename}")

    return {"received": data}
```

## 응답

```python
from flask import make_response, jsonify, redirect, url_for

@app.get("/")
def index():
    return jsonify({"hello": "world"})   # JSON

@app.get("/plain")
def plain():
    return "text response", 200, {"X-Custom": "yes"}

@app.get("/redirect")
def to_home():
    return redirect(url_for("index"))

@app.get("/custom")
def custom():
    resp = make_response({"ok": True})
    resp.set_cookie("session", "abc", httponly=True, secure=True)
    return resp
```

## Session

## Async (Flask 2.0+)

Flask 는 sync 지향이지만 `async def` view 를 제한적으로 지원:

```python
@app.get("/async")
async def async_view():
    await asyncio.sleep(0.1)
    return "done"
```

내부적으로 async view 를 sync 스레드에서 실행 (asgiref.sync.async_to_sync). **비동기 이점 없음**. 진짜 async 앱은 FastAPI, Quart, Starlette 사용.

## 개발 vs 프로덕션

```bash
# 개발
flask --app app run --debug

# 프로덕션 (Gunicorn)
gunicorn myapp:create_app() --workers 4 --bind 0.0.0.0:8000
```

**절대 `app.run(debug=True)` 로 프로덕션 배포 금지**. 내장 서버는 개발용, debug=True 는 임의 코드 실행 취약점.

## 최근 (Flask 3.x) 변화

- **Python 3.9+ 만 지원**
- **`route` 대신 method-specific 데코레이터** (`@app.get`, `@app.post`, ...)
- **Blueprint 의 `url_prefix` 오버라이드 개선**
- **`typing` 관련 개선**
- **DEBUG 모드 개선**
- **Werkzeug 3.x**: signature 검증 강화, HTTP/2 방향 준비 (실제 http/2 지원은 미정)

## Flask vs FastAPI

| 축 | Flask | FastAPI |
|:---|:---|:---|
| **비동기** | Sync 중심 (제한적 async) | Async first |
| **타입 검증** | 수동 (Flask-WTF, marshmallow) | 자동 (Pydantic) |
| **OpenAPI 자동** | X (Flask-Smorest 등 추가) | O |
| **성능 (async)** | 낮음 | 높음 |
| **생태계** | 20년, 방대 | 신생 (2018+), 성장 중 |
| **템플릿 (SSR)** | Jinja2 통합 | 별도 설치 필요 |
| **학습 곡선** | 완만 | 완만 |
| **관용** | Blueprint 파일 분할 | APIRouter |

**FastAPI 로 대체할 것인가**: 새 REST API 프로젝트는 FastAPI 유리. 서버-사이드 렌더링 (Jinja2) 중심 웹앱은 Flask 여전히 우세. 기존 Flask 서비스 마이그레이션은 비용 정당화 필요.

## 함정

> [!WARNING]
> **`app.run(debug=True)` 를 프로덕션에 노출 금지**. Werkzeug debugger 는 원격 코드 실행 취약. 반드시 Gunicorn/uWSGI.

> [!CAUTION]
> **thread-local 컨텍스트 함정**. request 스코프 데이터는 `g`. 함수 전역에 저장하면 request 간 leak.

> [!WARNING]
> **`async def` view 는 진짜 async 가 아님**. 성능 이점 없음. 진짜 async 필요하면 FastAPI/Quart.

> [!IMPORTANT]
> **Extension 초기화 순서**. `db.init_app(app)` 는 `create_app` 안에서. 앱 생성 시점 전에 하면 여러 앱 인스턴스 문제.

> [!CAUTION]
> **JSON 응답의 datetime, UUID**. Flask 3.x 는 자동 지원 확대, 하지만 일부 케이스 직접 처리 필요.
